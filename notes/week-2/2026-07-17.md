### 1. Moss PR #31 架构重构深度分析

PR #31 是 moss 最大的一次架构变更：+5396/-4199 行，126 个文件。核心是把"声明式安全"（Plan expects vs simulation effects）替换为"解析式安全"（Receipt parser 从有序 Changes 中解析结构化 Outcome）。

#### 关键术语变更

| 旧概念 | 新概念 | 说明 |
|--------|--------|------|
| Plan | Capability tree | 不再有独立的 Plan。可执行结构是 CapabilityNode 树，TransactionNode 是叶子 |
| Expects | 无 | 不再有"声明 vs 实际"的量化对账。安全性转移到 Receipt parser + Change 顺序验证 |
| Effects summary | Receipt tree | 模拟结果不再是资产进出汇总，而是 Receipt 递归结构 |
| Observation plane | Receipt.text | 不再有双平面。Receipt.text 是 presentation，Outcome 是权威数据 |
| Confirmation | ReceiptChange 身份验证 | 不再 confirmed observations。改为 ReceiptChange retain exact original Change object |
| Dealer | 无 | 预处理事件过滤/聚合机制移除 |
| Well-known token / Token table | TokenReference | 不再用 symbol 查找。TokenReference = address 或 "native" |
| Semantic type | Parameter type + declaration | `{ type: Zod, description: string }` 分离：context-free 值契约 + method-specific 用途 |
| Step builder | ProtocolRef dependency | 跨协议复用不再用独立函数。Protocol 声明 `protocols: { erc20: ERC20 }`，Registry 注入 typed instance |
| Manifest registration | Registry.use() scan exports | 不再有 defineProtocolPackage()。Registry 扫描模块 top-level exports，通过 PROTOCOL_META symbol 识别 |

#### 新三层安全机制

**第一层：编译时类型安全**
- `@Protocol`, `@Capability`, `@Query`, `@Receipt` 装饰器注入类型信息
- Handle 带 ABI 类型参数，所有方法调用有完整类型推断
- 每个 Capability 恰好一个 direct TransactionNode + 一个 named Receipt parser（框架验证）

**第二层：运行时结构验证**
- Registry.register() 验证：description 非空、category/verb/risk 合法、receipt 名称匹配 @Receipt 方法、params 有 Zod schema + description
- flattenCapabilityTree() 验证：每个 CapabilityNode 恰好一个 direct TransactionNode child
- verifyReceiptCoverage() 验证：ReceiptChange leaves 与 input Changes 精确 identity + length + order 匹配

**第三层：模拟时执行验证**
- createTraceSimulator() 逐交易执行：traceWithCalls → extractChanges → receipt parser → verifyReceiptCoverage
- 任何 Warning（REVERTED, TRACE_FAILED, CHANGE_ORDER_UNAVAILABLE, RECEIPT_FAILED, CHANGE_COVERAGE_MISMATCH, STATE_CHAIN_FAILED）立即 halt
- simulate() 返回 ordered Receipt leaf texts 给 MCP/Agent，SDK 保留完整的 Changes, Receipt trees, data, Outcomes

#### 新数据结构

**Capability tree：**
`CapabilityNode { kind, protocol, method, params: JsonSafeValue, receipt, children: [CapabilityNode|TransactionNode] }`
`TransactionNode { kind, transaction: { from, to, data: Hex, value: Hex } }`

**Change（模拟产生的原始证据）：**
`Event { kind: "event", address, topics: Hex[], data: Hex }`
`nativeTransfer { kind: "nativeTransfer", from, to, value: string }` // decimal string, not bigint

**Receipt（结构化解释）：**
`Receipt<TOutcome> { kind: "receipt", outcome: TOutcome, text: string, changes: (ReceiptChange | Receipt)[] }`
`ReceiptChange { kind: "change", change: Change (exact identity), data: JsonSafeValue, text: string }`

#### Kuru 适配器新写法要点

- swap Capability 先调用 `erc20.approve()`（嵌套 Capability），再调用 `router.anyToAnySwap()`（direct TransactionNode）
- 使用 `PositiveDecimalString` human token amounts，Protocol 内部 parseUnits 转换到 base units
- `#discoverRoutes()` 动态发现 verified markets，`#verifyMarket()` 链上验证
- `#quoteExactInput()` / `#quoteTargetOutput()` 二分搜索报价
- swapReceipt 解析 KuruRouterSwap event + Trade events，验证 taker 是 Router

#### ABI 管理流程

`update:abis (network)` → `abis-src/*.json + VENDOR.json` → `gen:abis (offline)` → `src/abis/kuru.ts`

- `update-abis.ts`：从 npm registry 拉取 SDK tarball（dist-tags.latest + 7 天 release-age guard），提取 abi/*.json 到 abis-src/，记录 VENDOR.json
- `gen-abis.ts`：离线从 abis-src/ 读取 JSON，生成 `src/abis/kuru.ts`（`export const XAbi = [...] as const`）
- `test/abis.test.ts`：断言 committed TS 等于 generate() 输出 byte-for-byte

### 2. Fetch-ABI CLI 实现

完成了 Issue #28 的实现：repository-level CLI，通过 Monadscan Etherscan V2 API 获取验证过的合约 ABI，输出确定性 TypeScript 代码。

#### 架构设计

`scripts/cli.ts`（薄包装入口）+ `scripts/fetch-abi.mjs`（核心逻辑库，导出 run(argv, env, deps)）+ `tests/fetch-abi.test.ts`（731 行测试）。CLI 与库分离。

**CLI 与库分离。** `cli.ts` 只负责 main() 调用和 process.exit()，`fetch-abi-logic.mjs` 导出纯函数 `run(argv, env, deps)` 接受依赖注入。测试直接 import 库文件 mock fetch/stdout/stderr，不 spawn 子进程。

#### 核心功能

- **API 调用** — Etherscan V2 (`chainid=143, module=contract, action=getabi`)
- **输入验证** — 地址格式 `^0x[0-9a-fA-F]{40}$`，导出名 TS 标识符校验
- **安全** — `redact(msg, key)` 防御性脱敏所有错误路径，API key 永不泄露
- **退出码** — 0=成功, 1=输入错误, 2=API/网络错误
- **Header 格式** — ADR 0007 explorer origin header，对齐列 + Endpoint 元数据
- **特性** — `--date YYYY-MM-DD` 锁定日期, `MONADSCAN_API_URL` 覆盖
- **零依赖** — Node 22 内置 fetch，tsx 运行

#### 测试覆盖（17/17 通过）

| 类别 | 测试数 | 覆盖内容 |

| CLI error paths | 3 | non-address arg、bad export name、fewer args |

#### 使用方式

CLI 直接读环境变量 `process.env.MONADSCAN_API_KEY`，不接收命令行参数。用法：先 `export MONADSCAN_API_KEY=<your-key>` 设置环境变量，然后 `pnpm fetch-abi <address> <ExportName> > packages/protocols/<name>/src/abis/<file>.ts` 输出到 stdout，shell 重定向写入文件，最后用 `git diff packages/protocols/<name>/src/abis/<file>.ts` 验证。示例（Kuru Router）：`export MONADSCAN_API_KEY=***; pnpm fetch-abi 0xd651346d7c789536ebf06dc72aE3C8502cd695CC KuruRouter > packages/protocols/kuru/src/abis/router.ts`。

## 疑问与思考

1. **Receipt parser 能检测到恶意协议吗？** ADR 提到 Moss 不声称检测同时控制交易编码和 Receipt 解析的恶意协议。安全边界在哪里？
2. **Receipt tree 嵌套委托边界在哪？** changesReceipt([change]) 可以委托给其他 Protocol 的 Receipt parser。连续 Change interval 如何界定？
3. **Parameter type 和 description 分离的实际好处？** 目前看只是文档层面分离——type 描述值的表示/单位/约束，description 描述字段在方法中的角色。对类型推断或运行时验证有影响吗？
4. **Registry.use() 扫描 exports 的性能？** 对于大型协议包，遍历所有 values 检查 PROTOCOL_META symbol 会不会成为注册时的性能瓶颈？
5. **Human token amounts 精度处理** — Protocol 内部 parseUnits 转换到 base units。这个转换是运行时还是编译时做的？
