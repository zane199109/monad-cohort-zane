**关键发现：**

1. **Moss 解决的问题** — AI Agent 与 DApp 交互时，手搓 calldata 风险极高（差一个 decimal 就可能丢钱）。Moss 把复杂性从 Agent 侧转移到协议适配层，Agent 只需说人话（"swap 1 MON for USDC"），Moss 负责组装交易、模拟验证、报告效果。

2. **核心安全机制** — `discover → load → action → simulate` 四步流程 + Effects 对账 + Halt Rule（有 warning 就停）。Moss 永不签名、永不发送，私钥始终在用户钱包。

3. **Issue 分析揭示的路线图** — 12 个 open issues 中 8 个是 adapter 请求（DEX/Lending/Yield），说明 Moss 正在从 Monad DEX 向全 DeFi 协议扩展。perpetuals 设计、ERC-1155/4626 接口层任务表明它瞄准的是多协议标准。

4. **与 PrismSettle 的关联** — Moss 的能力层抽象思路（人类可读参数 → 自动组装交易）与 PrismSettle 的 Registry + Reputation 体系有相似的设计哲学：都是降低 Agent 经济中的信任成本。

### Dev Plan 方向思考

Day 2 的核心任务是定义一个 AI-assisted MVP 原型。结合 PrismSettle 当前进度，有两个方向值得考虑：

| 方向 | 说明 | 难度 | Hackathon 价值 |
|------|------|------|----------------|
| A. Registry 合约 v1 | 实现 Registry 核心逻辑（注册、查询、基础 reputation 打分） | 中等 | 高 — 直接展示 infra 深度 |
| B. 256-shard 分片逻辑 demo | 模拟分片存储如何减少 Monad OCC 冲突 | 较低 | 中 — 偏理论，demo 效果好但合约层面简单 |

**初步倾向：** 方向 A 更适合作为 Week 2 的 Prototype Evidence，因为它：
- 有真实的智能合约产出
- 能展示 Foundry 测试能力
- 与 PrismSettle 主线直接相关
- 后续可扩展为完整的 Arbitration Module
