### 3. Moss 协议适配器模板包深度解析

分析了 `packages/protocols/_template/` 的完整结构和每个文件的作用：

| 文件 | 职责 |
|------|------|
| `src/index.ts` | 包的入口，只导出不注册 |
| `src/adapter.ts` | 核心：Protocol 类 + Capability + Query + Receipt |
| `src/abis/example.ts` | 合约 ABI（从链上获取，不手写） |
| `test/adapter.test.ts` | 运行时测试（tree 构建 + Receipt 解析） |
| `test/types.fixture.ts` | 编译期类型测试（@ts-expect-error 验证） |
| `package.json` | 包配置（private: true, workspace:* 依赖） |
| `vitest.config.ts` | esbuild 降级装饰器 + 源码别名 |

**核心设计哲学：**

- Protocol 通过顶层导出被发现，不手动注册
- 每个 Capability 恰好一个 TransactionNode + 一个 Receipt parser
- ABI 必须来自 compiled/explorer/vendored 三选一，绝不手写
- 类型安全是公共 API 行为的一部分

## 核心技术概念总结

### Moss 工作流四步

discover → load → action → simulate

- discover：查找操作坐标
- load：获取合约存根（intent/risk/schema）
- action：构建 Capability tree
- simulate：本地模拟执行，提取 Changes

### 安全机制

| 机制 | 说明 |
|------|------|
| **Warning = Halt** | 任何模拟警告立即停止，绝不继续 |
| **Receipt 穷尽验证** | 每个 Change 必须被 parser 覆盖 |
| **Intent Alignment** | 零 Warning 不等于结果正确，还需验证意图匹配 |
| **Moss 永不签名** | 私钥始终在用户钱包 |

### 类型与值导入区别

- **类型导入** — 只在编译时用，不产生运行时代码：`import { type MossRuntime } from "@themoss/core"`
- **具名导入** — 导入指定的几个导出：`import { NATIVE, Registry } from "@themoss/core"`
- **命名空间导入** — 打包所有导出为一个对象：`import * as erc from "@themoss/erc"`

## 疑问与思考

### 1. 为什么 Moss 用 `@themoss/*` 而不是 `@moss/*`？

作者 nishuzumi 在 commit `a83434a` 中把 npm scope 从 `@mossxyz` 改成了 `@themoss`。这是一个自定义的作用域名，不是 npm 约定的。

### 2. `.changeset/config.json` 的作用

管理 monorepo 的版本号和发布流程：

- `linked` — 这些包版本必须同步（core/simulator/erc/system/protocol-kuru/mcp-server）
- `fixed` — 空数组 = 每个包独立版本号
- `ignore` — 跳过这些包的发布（example 包不需要发布）

### 3. 协议适配器的标准姿势

每个新的协议适配器（PancakeSwap、Aave 等）都应该按照 `_template` 的结构来写：

1. 复制 `_template` 到新目录
2. 替换 `src/abis/` 中的 ABI
3. 实现 `@Protocol` 类 + Capability + Query + Receipt
4. 添加运行时测试和编译期类型测试
5. Monad 主网 happy path 必须零 Warnings
