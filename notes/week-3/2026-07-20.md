### 2. TypeScript 类型系统深度调试

**遇到的问题：**

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| `Type 'TokenRef' is not assignable to bigint` | TokenReference 是 Zod schema 不是类型 | 使用 `typeof TokenReference` |
| `Property 'balanceOf' does not exist on Handle` | ABI 包含 constructor/error/fallback 导致类型推断失败 | 创建 focused ABI 仅含 function+event |
| `'TokenReference' refers to a value, but is being used as a type` | 直接引用值而非类型 | 改用 `typeof TokenReference` |
| `Argument of type 'TransactionNode' is not assignable to bigint` | Handle 方法参数类型不匹配 | 检查 `parseUnits` 返回值与 ABI 输入类型对齐 |

**自己的理解：**

moss 的 Handle 类型系统要求 ABI 必须是纯 function+event 结构。如果 ABI 包含 constructor、error、fallback，TypeScript 无法正确推断方法签名。这是"最小信息原则"——类型推断只需要语义相关部分。

> Focused ABI 策略：将完整 ABI 拆分为两个文件。`fastlane.ts` 存放 ADR 0007 标注的完整 ABI（用于 runtime），`focused.ts` 存放去重后的 function+event ABI（用于 Handle 类型推断）。

## 关键收获

1. **Focused ABI 是必要模式** — Moss 的 Handle 类型推断对 ABI 纯净度要求极高，constructor/error/fallback 会污染类型系统
2. **TokenReference 必须用 typeof** — 它是 Zod schema 对象，不是 TypeScript 类型，直接引用会报错
3. **ADR 0007 不仅是标注** — ABI 来源分级影响信任链，explorer tier 需要区块浏览器验证记录
4. **Receipt parser 是安全核心** — 每个 Change 必须被恰好覆盖一次且顺序一致，这是防止 Agent 误解交易后果的最后防线
5. **PR 审查要读代码** — 从 Moss PR #100 x402 审查中学到：架构边界判断比功能实现更重要
