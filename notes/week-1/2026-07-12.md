## Day 1 — 职业方向选择

### 学习目标

- 理解 Web3 团队中 Research / Ops / Dev 三类角色分别解决什么问题
- 根据兴趣、能力、时间和 Week 3 组队目标，选择一个主职业方向
- 建立 Week 2 Role Log，用于记录学习材料、Prompt、链接、截图、错误、判断变化和每日产出
- 完成第一次 AI Collaboration Log，明确哪些工作可以交给 AI，哪些必须由人类判断和核查

### 角色定义

一个完整的 Web3 产品或项目不只需要开发者，通常需要三类 Builder 共同协作：

- **Research Builder**：发现问题，理解协议、产品、治理、生态机会和风险
- **Ops Builder**：连接用户，设计社区运营、Space / 小型活动策划、传播执行和反馈复盘机制
- **Dev Builder**：解决问题，把需求变成可运行的原型、工具、合约、脚本或 Agent 能力

## 核心工作流原则

Dev Track 的核心 AI 协作模式：

> AI 读文档 → 拆任务 → 生成代码骨架 → 解释错误
> 人类负责运行 → 修改 → 验证 → 记录

必须记录 AI Collaboration Log：
- AI 帮了什么
- 人类删改/核查了什么
- 哪些不能交给 AI

Prototype 必须有可检查证据：
- repo + README
- 截图/录屏
- 交易 hash（Monad testnet 部署）或明确的 mock 边界
- 错误日志
- 人工修改记录

## 总结

Week 2 是 Builder 从"统一学习"转向"分方向深耕"的关键过渡周。核心逻辑：

1. **Day 1 定方向**：不是所有人都做一样的事，Dev 方向的任务是选准一个可验证的产出物，围绕它展开。
2. **Day 2-3 做执行**：从 Dev Plan → Code Skeleton，逐步把想法变成可运行的原型。AI 负责读文档、拆任务、生成骨架；人类负责运行、修改、验证、记录。
3. **Day 4 出证据**：原型必须有可检查的证据——repo、README、截图/录屏、交易 hash 或 mock 边界。
4. **Day 5 收 Portfolio**：汇总所有产出，形成 Week 3 组队的筹码。

对 Dev 方向而言，Week 2 的本质不是学新技术，而是**建立一套可复现的开发 + AI 协作 + 证据记录的工作流**。这份流程本身，就是 Week 3 和后续 Hackathon 中最有价值的资产。
