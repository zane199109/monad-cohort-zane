## 学习内容

### 1. AI 辅助合约开发：从 Prompt 到人工审查

**核心流程：** 写 Prompt → AI 生成初稿 → 我阅读 + 让 AI 解释 → 人工检查 → 记录修改 / 判断。

**Prompt 设计思路：**
- 明确要求用 OpenZeppelin 标准库，避免 AI 从零实现 ERC721
- 明确"保持简洁、安全，不要过度设计"，抑制 AI 顺手加白名单、销毁、可升级代理等超出需求的特性
- 需求点刻意列得最小：只覆盖 创建类型 / 铸造 / 查询 三个核心能力

**AI 初稿的 6 个问题及我的修改：**

| # | AI 初稿问题 | 我的修改 | 原因 |
|---|-----------|---------|------|
| 1 | `mint` 用 `_mint` | 改为 `_safeMint` | `_mint` 不检查接收方是否为合约，铸造给合约会导致代币永久卡死 |
| 2 | `hasBadge` 试图用 `balanceOf` + 循环枚举，但没有 tokenId→typeId 映射，逻辑跑不通 | 新增 `tokenBadgeType` 和 `badgeCount` 两个映射，hasBadge 改为 O(1) 查 `badgeCount` | AI 这段是"看起来对、实际无法实现"的典型 |
| 3 | `mint` 没校验 typeId 是否存在 | 增加 `BadgeTypeNotFound` revert | 防止铸造幽灵类型 |
| 4 | `createBadgeType` 没校验 name 非空 | 增加 `EmptyName` revert | 避免无意义空名类型 |
| 5 | 用 `require("...")` 字符串报错 | 改为 custom error | 省 gas + 更结构化 |
| 6 | 测试用 `vm.expectRevert("Ownable: caller is not the owner")` | OZ v5 已改用 custom error，需 `abi.encodeWithSelector` 完整匹配 | 初版测试 3 个全挂，修后 5 个全过 |

**刻意拒绝的过度设计：** burn、batchMint、可升级代理、白名单/领取价格

**合约架构（审查后定稿）：**

> `NFTBadge is ERC721URIStorage, Ownable`
> `├── struct BadgeType { name, description, uri, exists }`
> `├── mapping(uint256 => BadgeType) badgeTypes`
> `├── mapping(address => bool) minters`
> `├── mapping(uint256 => uint256) tokenBadgeType  // tokenId → typeId`
> `├── mapping(address => mapping(uint256 => uint256)) badgeCount  // owner → typeId → count`
> `├── uint256 nextTokenId`
> `├── uint256 nextBadgeTypeId`
> `├── createBadgeType(string, string, string) onlyOwner`
> `├── setMinter(address, bool) onlyOwner`
> `├── mint(address, uint256) onlyMinter`
> `├── hasBadge(address, uint256) view → bool  // O(1)`
> `├── getBadgeType(uint256) view → BadgeType`
> `└── events: BadgeTypeCreated, BadgeMinted, MinterUpdated`

### 4. 前端交互：从空白页面到完整 Badge Wall

**问题 1：Tailwind CDN 国内不通 → 页面空白**
- 现象：`<script src="https://cdn.tailwindcss.com">` 加载失败
- 解决：用 Tailwind CLI 静态生成 `output.css`（28KB），替换 CDN 引用

**问题 2：ethers.js CDN 也可能不通**
- 解决：本地下载 `ethers.umd.min.js`（760KB）放到 frontend 目录

**问题 3：`wallet_requestPermissions` 返回账户为空**
- 现象：连接钱包后日志无任何输出，页面仍显示"connect wallet"
- 原因：MetaMask 返回的 `allowedAccounts` 数组格式与预期不符，fallback 也没拿到
- 解决：重构账户获取逻辑，先尝试解析 `perms[0].caveats`，失败后 fallback 到 `eth_accounts` 直接获取

**问题 4：mint typeId=2 时报 `BadgeTypeNotFound(2)`**
- 现象：点击 typeId=2 的 badge 时 revert，错误选择器 `0xb62d7eb2`
- 原因：链上只创建了 typeId 0 和 1，mock-data 定义的 12 种类型未全部创建
- 解决：批量创建 typeId 2-11（见上文 §3）

**前端功能：**
- 连接 MetaMask → 自动检测/切换 Monad Testnet
- Badge Wall 展示 12 种徽章
- 点击 badge 查看详情 → Mint 铸造
- 右侧日志抽屉记录完整操作链（含可点击跳转 Explorer 的 txHash）

## 关键收获

1. **AI 能快速给结构骨架，但安全细节和"能不能跑通"必须人工把关。** `_safeMint` vs `_mint`、`hasBadge` 的映射缺失都是 AI 看不出的问题
2. **依赖库版本差异是隐蔽坑。** OpenZeppelin v5 把 `Ownable` 报错从字符串改成 custom error，照搬旧写法测试全挂——只有实际跑测试才暴露
3. **"keep it simple" 要人来守底线。** AI 容易顺手加一堆没要求的特性，得明确拒绝
4. **真正可信的检查点是跑出来的。** 编译通过、测试通过，比"看着像对的"有说服力得多
5. **前端与链上状态必须对齐。** mock-data 定义了 12 个 badge，但链上只创建了 2 个类型，mint 会 revert。部署后需要批量补全
6. **Foundry 比 Remix 更适合工程化开发。** 编译、测试、部署一体化，且支持完整的测试框架（`forge test`），比 Remix 的在线环境更高效
7. **国内网络环境下 CDN 不可靠。** Tailwind CDN + ethers.js CDN 都可能不通，静态化是更稳妥的方案
8. **MetaMask 的 `wallet_requestPermissions` 返回值格式不固定。** 需要多种 fallback 策略
