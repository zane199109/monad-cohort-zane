## 设计调整记录

### 1. 钱包断连后无法切换账号

**现象：** 用户断开连接后，点击 Connect Wallet，MetaMask 直接返回已授权的账户，不弹出账户选择窗口。用户根本没有机会切换。

**技术分析：**

钱包连接流程经过三次迭代：

| 方案 | 做了什么 | 为什么不行 |
|------|---------|-----------|
| 第一次尝试：`eth_requestAccounts` | 每次连接都请求授权 | 对已授权站点直接静默返回已授权账户，不弹窗 |
| 第二次尝试：`wallet_requestPermissions` | 强制重新请求 eth_accounts 权限 | 大部分 MetaMask 版本同样静默通过，不弹窗 |
| 第三次尝试：`wallet_requestPermissions` 条件分支 | 断连后标记 `wasDisconnected=true`，触发 `wallet_requestPermissions`；刷新页面用 `eth_requestAccounts` | 同第二版，弹窗行为不统一 |

**最终方案：`wallet_revokePermissions` 主动撤回授权**

> ```
> // disconnect() 中
> window.ethereum.request({
>   method: "wallet_revokePermissions",
>   params: [{ eth_accounts: {} }],
> }).catch(() => { /* 旧版 MetaMask 忽略 */ });
> 
> // connect() 中直接使用 eth_requestAccounts
> const accs = await window.ethereum.request({ method: "eth_requestAccounts" });
> ```

**设计思考：** 之前一直想的是"怎么在连接时让 MetaMask 重新选账户"，但 MetaMask 没有提供可靠的编程式账户切换接口。换个思路——**断开时真正撤回授权**，下次连接时 `eth_requestAccounts` 自然就会弹出选择窗口。刷新页面不影响，因为 MetaMask 还会记住之前的授权。

### 2. accountsChanged 事件导致连锁问题

**现象：** 用户已连接 → 点击 Connect 按钮 → MetaMask 弹出确认 → `accountsChanged` 事件触发 → 旧代码 `location.reload()` 刷新页面 → 状态全丢 → 需要再点一次 Connect。

**修复：**

> ```
> // 改前
> window.ethereum.on("accountsChanged", () => location.reload());
> 
> // 改后
> window.ethereum.on("accountsChanged", (accounts) => {
>   if (accounts.length === 0) disconnect();
>   else connect(true);  // force-reconnect，走 restore 路径
> });
> window.ethereum.on("chainChanged", () => connect(true));
> ```

**设计思考：** `location.reload()` 是早期开发时图省事的写法——"反正刷新后一切重来"。但在链上 DApp 中，这种偷懒会破坏用户体验：MetaMask 连接成功后触发的事件不应该让页面白闪一下。

### 3. 前端 8080 + 后端 8082 双端口架构与 CORS

**现象：** Admin 页面可以打开，但 API 调用全部 404。前端 `npx serve` 无 API 路由，API 请求走相对路径 `/api/claims/...` 落在了 8080 上。

**双端口架构：**

| 端口 | 服务 | 内容 |
|------|------|------|
| 8080 | `npx serve`（静态文件） | HTML, CSS, JS |
| 8082 | Go 后端 | API `/api/claims/*` |

**修复：**

1. 前端代码中引入 `API_BASE` 常量指向 8082

> ```
> const API_BASE = "http://localhost:8082/api/claims";
> // 所有 fetch 调用都使用 API_BASE 而非相对路径
> ```

2. Go 后端添加 CORS 中间件

> ```
> w.Header().Set("Access-Control-Allow-Origin", "*")
> w.Header().Set("Access-Control-Allow-Methods", "GET, POST, OPTIONS")
> w.Header().Set("Access-Control-Allow-Headers", "Content-Type")
> ```

3. Admin 页面从 `backend/web/admin/` 复制到 `frontend/admin/`，`npx serve` 可直接访问

**设计思考：** 开发时双端口分离是标准实践——前端 dev server 负责热更新和静态文件，后端专注 API。但不解决 CORS 就容易出问题。之前 Go 后端直接托管前端不需要 CORS，用 `npx serve` 后就需要了。两种方式应该都支持。

### 4. Claim 提交流程设计

**现象：** 用户提交 claim 后，modal 关闭，但徽章卡片仍然显示 "Submit Claim" 按钮。用户可以多次提交同一徽章的 claim。

**设计状态机：**

> ```
> ┌─────────┐   submit    ┌─────────┐   admin approve   ┌─────────┐
> │ No Claim │ ────────→ │  Pending  │ ──────────────→  │  Minted  │
> │（可提交） │           │（不可提交）│                  │（不可提交）│
> └─────────┘           └─────────┘                   └─────────┘
>                            │  admin reject
>                            ↓
>                         ┌─────────┐
>                         │ Rejected │  ← 可重新提交（Resubmit）
>                         └─────────┘
> ```

**实现：**

1. 后端新增 `GET /api/claims/check?user_addr=X&type_id=Y` 接口，返回该用户对该 badge 的最新 claim 状态
2. 前端 `state.claimStatus[typeId]` 缓存每个 badge 的 claim 状态
3. 提交成功后直接设置 `state.claimStatus[claimTypeId] = 'pending'` 并重渲染
4. 卡片按钮根据状态显示不同文案和行为：
   - `""`（无 claim）→ "Submit Claim"
   - `"pending"` → 禁用按钮显示 "Pending Review"
   - `"rejected"` → "Resubmit Claim"
   - `"minted"` → 无按钮（badge 已解锁）
5. 连接钱包时自动调用 `refreshClaimStatuses()` 从后端拉取所有 badge 的 claim 状态

### 5. Go fmt.Sscanf 匹配缺陷

**现象：** Admin 面板点击 reject 按钮报错 "invalid claim ID"。

**根因：**

> ```
> // 问题代码：
> n, err := fmt.Sscanf(r.URL.Path, "/api/claims/%d/approve", &id)
> if n == 0 {  // ← 这里只检查 n==0
>     n, err = fmt.Sscanf(r.URL.Path, "/api/claims/%d/reject", &id)
>     action = "reject"
> } else {
>     action = "approve"
> }
> ```

当路径是 `/api/claims/3/reject` 时：
- 第一次 Sscanf 匹配 `/api/claims/%d/approve`：`%d` 成功匹配了数字 `3`（n=1），但后续 `/approve` vs `/reject` 不匹配
- Sscanf 返回 `n=1, err="input does not match format"`
- 因为 `n==0` 判断为假，代码错误地走了 `else` 分支，设 `action="approve"`
- 然后在 approve 逻辑中，合约调用 `mint` 时报错

**修复：** `if n == 0 || err != nil`

### 6. 断开连接时隐藏 Admin 链接

**现象：** 用户断开钱包后，导航栏的 Admin 链接仍然可见。只有 `checkAdminRole()` 用于控制显示/隐藏，但断开时该方法有 `if (!state.account) return;` 的 early return，不会执行隐藏逻辑。

**修复：** 在 `disconnect()` 中直接隐藏：

> ```
> const adminLink = $("adminLink");
> if (adminLink) adminLink.classList.add("hidden");
> ```

## 关键收获

1. **wallet_revokePermissions 是断连切换账号的最可靠方案。** wallet_requestPermissions 和 wasDisconnected 条件分支都不稳定，只有真正撤回授权才能让下一次 eth_requestAccounts 弹出选择窗口。

2. **架构选择要考虑开发体验和测试体验的平衡。** 双端口（8080+8082）对开发友好，但需要处理好 CORS；单端口（Go 直接托管前端）对测试友好，但改前端代码要重启后端。两种方式并存，各有利弊。

3. **状态机驱动的 UI 比条件判断更可靠。** Claim 流程有四态（无 → pending → minted / rejected），用显式的状态枚举 + 状态机转换比一堆 if-else 判断更清晰，也更容易维护。
