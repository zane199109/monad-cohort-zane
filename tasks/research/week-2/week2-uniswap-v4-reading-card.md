# Product / Protocol Reading Card: Uniswap v4

---

## 用大白话解释：Uniswap v4 到底是什么？

### 先说说 Uniswap 本身

想象你是一个小贩，在集市上摆摊卖水果。你标价"1 个苹果 = 1 个橙子"，有人拿苹果来就给他橙子，拿橙子来就给他苹果。你赚的是每笔交易的"摊位费"（手续费）。这就是 **Uniswap v1/v2** —— 简单粗暴，谁都能摆摊。

但问题是，每个水果摊都是独立的一个棚子，你要卖新品种就得搭新棚子，费时费力。

### v3 升级了什么？

后来集市升级了，允许你在摊位上只放"最好卖的货架位置"（集中流动性）。以前 100 个苹果得摆满整张桌子，现在可以只放 10 个在黄金位置，剩下的放仓库。**资本效率提高了**，但缺点是每个新水果品种还得单独搭一个棚子，而且你想搞点"买二送一"之类的促销活动？没门 —— 规则是固定的，改不了。

### v4 干了件大事

v4 做了三件事，把集市彻底改造了：

**① 一个大厅，代替一百个小棚子（Singleton）**

以前每对交易对（ETH/USDC、ETH/WBTC...）都要搭建独立的棚子。v4 把所有人请进**一个大厅**，里面有个大账本写着"谁在哪个窗口放了什么、多少钱"。新开一个交易对？不再需要搭棚子，只需在大厅角落加个窗口，成本从搭一个棚子变成**画一张纸**。Gas 费降了 90% 以上。

**② 先拿货、出门再付钱（Flash Accounting）**

假设你想用 A 换 B，中间经过 C、D 两个中间币。以前是你每到一个窗口就得交一次钱 —— A→C 付一笔、C→D 付一笔、D→B 再付一笔，每次都要掏钱包。v4 改成：你推着购物车走，每个窗口先记账（"拿了 A，欠 C"），最后到收银台一次性算总账 —— **中间一分钱不用付**，只付最终的差额。Gas 又省一大截。

**③ 给自动售货机装了"插件槽"（Hooks）**

这是最革命性的变化。

v3 的自动售货机是固定的：投币 → 掉饮料。v4 的自动售货机有 **8 个回调点**（before/after Initialize、AddLiquidity、RemoveLiquidity、Swap），你可以塞进自定义程序：

- **动态定价插件**：晚上 8 点后涨价 20%（因为夜宵需求大）
- **限价单插件**：只有比特币跌到 $50,000 以下才帮你买入（像超市限时特价抢购）
- **批量拆单插件**：有人一次性要买 1000 瓶可乐，机器自动分成 100 份慢慢卖，防止价格被冲垮（就像大客户买米，粮店不会一次性全倒给他，而是分批发货）
- **会员积分插件**：每交易一次攒积分，积分换手续费折扣

这些"插件"不是 Uniswap 官方写的 —— **任何人都可以自己写一个插件挂上去**。开发者不需要从零搭建一个交易所，只需要写一个几百行的小插件，塞进 v4 的自动售货机里就能跑。

### 打个比方总结

> v2 像一个**菜市场** —— 每个摊位各自为政，简单但低效。
> v3 像一个**精品超市** —— 货架利用率高了，但每个品牌还是独立的专柜。
> v4 像一个**装了智能操作系统的便利店** —— 所有货品统一管理，而且允许任何人在收银机上装 App 插件，想怎么卖就怎么卖。

## 解决的问题

Uniswap v3 的**集中流动性**（Concentrated Liquidity）在资本效率上取得了巨大突破，但存在三个结构性痛点：

1. **高部署成本** —— v3 中每个池子（pool）都是一份独立合约，每创建一个新池子（包括新的手续费等级、新的预言机配置）都需要部署新合约，Gas 成本高且碎片化。
2. **缺乏可编程性** —— v3 的 AMM 逻辑完全固化，无法在 swap 前后执行自定义逻辑。LP 只能提供流动性，不能附加任何策略（如动态手续费、TWAP 订单、限价单等）。
3. **WETH 包装负担** —— 用户每次用 ETH 交易都需要先 wrap 成 WETH，增加一步操作和 Gas。

Uniswap v4 针对这些问题重新架构，在保留 v3 集中流动性的资本效率优势基础上，引入**Singleton（单例合约）** 和 **Hooks（钩子机制）**，让 AMM 变得可编程。

## 核心机制

### 1. Singleton 架构

所有池子共存于同一份合约中，而不是每池一份独立合约。

- 这是通过一个内部账本（`PoolId → PoolState` 映射）实现的
- 创建新池子的 Gas 从 ~350K（v3）降至 **~30K**（v4）
- 跨池原子操作成为可能：同一笔交易内可以在多个池子间路由，只需在最后结算净差额

### 2. Flash Accounting（闪存记账）

用内部净额结算取代 v3 的即时 Token 转移。

- 在 `swap` 或 `modifyLiquidity` 调用期间，只记录仓位变化（debit/credit），不实际转移 Token
- 最后通过 `settle()` / `take()` 一次性结算净差额
- 这允许链上路由（multi-hop swap）仅转移一次"净差值"，大幅降低中间 Token 转移的 Gas

### 3. Hooks（钩子）

这是 v4 的核心创新 —— 用户在池子创建时注册一个 Hook 合约，该合约可以在以下 8 个关键点执行自定义逻辑：

| Hook 回调 | 触发时机 |
|---|---|
| `beforeInitialize` / `afterInitialize` | 池子初始化前后 |
| `beforeAddLiquidity` / `afterAddLiquidity` | 添加流动性前后 |
| `beforeRemoveLiquidity` / `afterRemoveLiquidity` | 移除流动性前后 |
| `beforeSwap` / `afterSwap` | 交易前后 |

通过 Hooks 可以实现：

- **动态手续费** —— 根据波动率或利用率调整费率
- **TWAMM（时间加权平均做市商）** —— 将大订单拆分成多个区块执行
- **链上限价单** —— Hook 自动触发 swap
- **Geyser / 挖矿奖励** —— 自动分配激励
- **预言机集成** —— 自定义喂价逻辑
- **MEV 保护** —— 批量处理或延迟交易

Hook 的实现不依赖许可，任何开发者都可以部署 —— 这种无许可的可编程性是 v4 最重大的设计突破。

### 4. Native ETH 支持

v4 原生支持 ETH，不再需要 WETH 包装。LP 可以直接提供 ETH 作为流动性，用户可以直接用 ETH 交易。

### 5. 自定义手续费等级

v3 只允许有限的手续费档位（0.05%、0.30%、1.00% 等），v4 允许通过 Hook 动态设置手续费，不再受限于预设档位。

## 主要用户

| 用户类型 | 价值定位 |
|---|---|
| **流动性提供者（LPs）** | 通过 Hook 实现自动化策略（动态费率、TWAMM 订单拆分），不再只是被动提供流动性 |
| **交易者 / Swap 用户** | 更低的 Gas（Singleton + Flash Accounting），原生 ETH 支持，更丰富的流动性选择 |
| **DeFi 开发者 / 协议** | 无需 fork 整个 AMM —— 只需写一个 Hook 合约即可定制 AMM 行为，极大降低创新成本 |
| **做市商 / 机构** | 可以部署自己的 Hook 实现私有做市策略，同时共享 v4 的流动性和路由网络 |

## 一个风险或疑问

**Hooks 的安全风险与采用率的矛盾。**

Hooks 是 v4 最具创新性的设计，但也带来了根本性挑战：

- **安全风险**：每个 Hook 都是一份独立合约，有独立的攻击面。LP 把资金注入一个使用 Hook 的池子时，实际上在信任两件事：(1) v4 合约本身安全，(2) Hook 合约安全。历史上 Aave 的 flash loan 攻击、Curve vyper 漏洞等事件表明，DeFi 的组合风险一旦叠加就会变得极难审计。v4 的 Hook 像是一个"插件市场"——如果某个流行 Hook 出现漏洞，整个依赖它的池子都会受影响。

- **采用率的疑问**：大多数 LP 是否真的需要或理解 Hooks？v3 的集中流动性本身已经让散户 LP 望而却步（需要主动管理价格区间），v4 再加上程序化策略门槛，可能进一步把普通 LP 推向被动策略平台（如 Yearn、Beefy）。如果 v4 的流动性主要由机构 / 做市商通过自定义 Hook 提供，那它是否实际上变成了一个"机构级 AMM 基础设施"，而不是 v2 时代的"人人可做市"？

我的疑问是：**Hooks 带来的灵活性提升是否足以抵消其增加的复杂性和安全风险，从而被主流 DeFi 用户（而非只是开发者）采用？** 或者说，v4 的真正用户其实是部署 Hook 的开发者协议，而不是直接使用它的终端用户？
